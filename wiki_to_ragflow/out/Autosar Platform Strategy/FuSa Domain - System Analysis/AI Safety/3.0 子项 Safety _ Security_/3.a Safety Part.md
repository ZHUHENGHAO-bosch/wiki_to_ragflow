### 0. 先把边界划清楚

#### 0.1 一张表看透 Security vs Safety

| 维度 | Security (信息安全) | Safety (功能安全 / 安全性) |
| --- | --- | --- |
| 保护谁？ | 系统 → 免受攻击 | 人/环境 → 免受系统伤害 |
| 谁是威胁？ | 恶意攻击者 (有意图) | 随机失效 + 罕见场景 (无意图) |
| 典型问题 | Prompt 注入、越权、数据泄露 | 幻觉导致错误车控、边缘场景识别失败 |
| 核心标准 | ISO/SAE 21434, UN R155, OWASP, MITRE | ISO 26262, ISO 21448, ISO PAS 8800 |
| 分级体系 | CVSS, 风险等级 | ASIL (A < B < C < D) |
| 验证方法 | 渗透测试、红队 | HARA、FTA、FMEA、STPA、场景测试 |
| 失效的"样子" | "有人黑进来了" | "它自己做错了" |

#### 0.2 两者的关系 —— 不是对立，而是 正交的双轴

```
                  ▲ Safety 维度 (纵坐标)
        高 Safety │ ① 完美系统          │
                  │  (两者都达标)        │
                  │                     │
        低 Safety │ ③ 危险且易攻击       │
                  └─────────────────────▶ Security 维度
                    高 Security   低 Security
                    
   ① 两者都要
   ② 安全但易被攻击 (能开车, 但随便谁都能劫持)
   ③ 既不安全也易被攻击
   ④ 难被攻击但自己开车就撞 (Security 满分 Safety 零分)
```

关键点 : Security 过关 ≠ Safety 过关 。LLM Agent 最大的风险反而在 Safety — 它即使没被任何人攻击，也可能因为 幻觉 / 训练分布外 / 置信度误估 而做出危险动作。

#### 0.3 一个例子直观感受差异

| 场景 | 性质 |
| --- | --- |
| 黑客通过 POI 描述注入, 让 Agent 开天窗 | Security (T-101 间接注入) |
| Agent 因训练数据偏差, 把驾驶员说的"关窗"幻觉成"开窗" | Safety (AI 功能不足) |
| 黑客伪造 Capability Token 开车门 | Security (T-306) |
| Agent 在大雾天气置信度虚高, 给出错误导航建议 | Safety (OOD/SOTIF) |
| 黑客篡改 OTA 包植入后门 | Security (T-104) |
| 模型更新后 漂移 , 老用户的习惯被误识别 | Safety (概念漂移) |

### 1. AI Safety 里 "Safety" 的三大支柱

ISO PAS 8800 的 Safety 视角是 ISO 26262 (FuSa) + ISO 21448 (SOTIF) + AI 特殊性 的组合：

```
┌────────────────────────────────────────────────────────────┐
│                  AI System Safety                          │
│                                                            │
│  ┌──────────────┐   ┌──────────────┐   ┌──────────────┐   │
│  │  Functional  │   │    SOTIF     │   │ AI-specific  │   │
│  │   Safety     │   │              │   │              │   │
│  │  (ISO 26262) │   │ (ISO 21448)  │   │(ISO 8800 新增)│   │
│  ├──────────────┤   ├──────────────┤   ├──────────────┤   │
│  │ HW 随机故障  │   │ 功能不足     │   │ 数据集缺陷   │   │
│  │ SW 系统性故障│   │ 触发条件     │   │ 模型局限     │   │
│  │ FTA/FMEA     │   │ ODD 边界     │   │ 概念漂移     │   │
│  │ ASIL 分级    │   │ 场景枚举     │   │ OOD 检测     │   │
│  │              │   │              │   │ 可解释/对齐  │   │
│  └──────────────┘   └──────────────┘   └──────────────┘   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 支柱 A: Functional Safety (传统功能安全 ISO 26262)

管的是 : 系统故障（硬件随机/老化 + 软件系统性 bug）不造成伤害。

AI 场景里的关键点：

- 推理硬件的随机失效 : NPU/GPU 位翻转对 Tensor 计算结果的放大效应
- ECC / Lock-step CPU : 车规芯片的传统保护依然适用
- 软件缺陷 : 推理框架、Tokenizer、decoding 代码的 bug
- 安全机制覆盖率 : Safety mechanism 要求 SPFM / LFM 指标

这部分 AI 和传统软件基本没差别 ，照 ISO 26262 做就行。

#### 支柱 B: SOTIF (预期功能安全 ISO 21448)

管的是 : 系统 没故障，但功能本身就不够好 ，在罕见场景下造成伤害。

核心概念：

| 概念 | 含义 | LLM Agent 例子 |
| --- | --- | --- |
| 功能不足 (Functional Insufficiency) | 规范不全 or 性能局限 | LLM 对方言识别率低；对新出的车型术语不认识 |
| 触发条件 (Triggering Condition) | 激活功能不足的外部条件 | 多人同时说话；背景有电视声；有口音 |
| ODD (Operational Design Domain) | 允许运行的场景边界 | 停车时允许所有 Tool；高速只允许娱乐 Tool |
| 危险事件 (Hazardous Event) | 因功能不足导致的整车级危险 | Agent 误解 → 误操作 → 影响驾驶 |

SOTIF 4 区域模型 ：

```
                    已知              未知
              ┌──────────────┬──────────────┐
      不安全  │  区域 2       │  区域 3       │
              │  已知不安全   │  未知不安全   │  ← SOTIF 工作重点:
              │  (必须消除)   │  (必须探索)   │    把 3 和 2 变成 1 或 4
              ├──────────────┼──────────────┤
      安全    │  区域 1       │  区域 4       │
              │  已知安全     │  未知安全     │
              └──────────────┴──────────────┘
```

SOTIF 的工程动作：

1. 场景库建设 （Scenario Catalog）—— 把潜在触发条件穷举出来
2. 场景覆盖测试 —— 用仿真 + 真实路测覆盖场景库
3. 置信度管理 —— 低置信时主动拒答 / 降级
4. OOD 检测 —— 识别分布外输入
5. 反馈闭环 —— 现场数据回流，迭代模型

#### 支柱 C: AI 特殊性 (ISO PAS 8800 新增)

管的是 : 传统 FuSa + SOTIF 还盖不住的 AI 独有问题。

这才是 "AI Safety 里 Safety" 最新颖的部分：

### 2. 9 个"Safety 相关的 AI 属性"（ISO PAS 8800 附录 D）★

这是标准里最值得抄作业的核心，每一条都是 Safety（ 不是 Security ）工程任务：

| # | 属性 | 定义 | 在 LLM Agent 里的具体要求 |
| --- | --- | --- | --- |
| 1 | Robustness 鲁棒性 | 输入扰动下输出稳定 | 同义改述、方言、口齿不清时仍给同样决策 |
| 2 | Generalization 泛化 | 未见过的输入也合理处理 | 新 POI / 新车型术语不崩 |
| 3 | Reliability 可靠性 | 长时间运行性能稳定 | 连续会话 N 小时不漂移 |
| 4 | Resilience 韧性 | 异常后能恢复 | 某 Tool 失败后 Agent 能重试/换路 |
| 5 | Controllability 可控性 | 人类能干预/叫停 | HITL、紧急停止、语音打断 |
| 6 | Predictability 可预测性 | 相同输入给相似输出 | 避免随机性 (temperature 收紧) |
| 7 | Explainability 可解释性 | 能说清为什么这么做 | Chain-of-Thought 可审计, 决策摘要 |
| 8 | Alignment 对齐 | 行为符合人类意图 | 不越权；按用户真实目的办事 |
| 9 | Fairness 公平性 | 不同群体一视同仁 | 不因性别/方言/地域差别对待 |

这 9 条和 Security 几乎无关，但全是 Safety 的核心 。ESW2 目前的任务清单完全没有覆盖这 9 条，这是真正的差距。

### 5. 具体到 LLM Agent 的 Safety 议题（最容易被忽视）

LLM Agent 特有、Security 根本覆盖不到的 Safety 问题：

#### 5.1 幻觉 (Hallucination) —— #1 Safety 风险

| 种类 | 例子 | 后果 |
| --- | --- | --- |
| 事实幻觉 | 把"充电桩"识别成"加油站" | 开错地方 |
| 工具幻觉 | 调用不存在的 Tool | 静默失败 |
| 参数幻觉 | 正确 Tool, 错误参数 | 误操作 |
| 推理幻觉 | 逻辑链条错误但结论"看上去对" | 难审计 |
| 法规幻觉 | 编造交规条款 | 违规 |

Safety 对策 （非 Security）：

- Self-consistency : 同一问题多次采样, 投票
- Retrieval-Augmented : 用可信知识库约束
- Tool-Schema Constrained Decoding : 硬约束输出格式
- 置信度 + 拒答 : 低于阈值不答
- 符号校验 : 用规则引擎做事后校验

#### 5.2 Overreliance (司机盲从)

学术研究显示司机对 LLM 建议的 盲批准率高达 85% 。这是纯 Safety 问题：

- HMI 设计上显式标注"AI 建议, 请自行判断"
- 关键决策引入 认知摩擦 (多停 0.5 秒让司机思考)
- 一致性测试 (同一问题反复问, 给相同/相似答案降低司机警觉)

#### 5.3 反直觉现象： 过度安全也是不安全

```
拒答率 低 ─────────────────────────── 高
         ↑                         ↑
    危险 (乱答)               烦躁 (啥都不干)
    司机被误导                司机关闭 AI
                             回到人工操作
                             可能更危险
```

Goldilocks 带 : 拒答率需要有 定量的最优区 ，不是越高越好。

#### 5.4 Reasonably Foreseeable Indirect Misuse (RFIM)

ISO 8800 3.3.8 明确要求覆盖的"可合理预见的间接误用"：

- 小孩坐驾驶位说"开车"
- 司机开玩笑说"自己开吧"
- 乘客装驾驶员来触发权限
- 司机喝了酒说话含糊

这不是攻击，是 真实误用 。Safety 要处理，Security 不管。

#### 5.5 Automation Complacency (自动化麻痹)

司机长期依赖 AI 后，自己的驾驶能力退化。这是 长期 Safety 问题：

- 设计上保持 司机动手动眼 (periodic re-engagement)
- 监测驾驶员警觉度 (DMS)
