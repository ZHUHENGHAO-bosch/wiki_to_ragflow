### Context Change:

Context change to different core on invocating an API will have an Overhead

Example Use case:

We decided to move Module B to Core 1 to improve performance

![](../../../_images/10.%20Challenges%20in%20Multicore/Multicore_Context_Change_UseCase.bmp)

So to call Module B in Core 1 from Module 0 there will be context change.

Context Change can be implemented in two ways

- Data driven invocation
- Event driven invocation

Both the implementation adds execution time overhead

- On the caller side (notification / serialization, shared buffer handling)
- On the called core (worker execution, task scheduling, deserialization)

> **INFO: Relocation decision**
> Relocation of an module should be done only when the module has an high execution time or small execution time but had to run more frequently In other words: There is not much sense in relocating a  small execution time which causes overhead that is larger

![](../../../_images/10.%20Challenges%20in%20Multicore/Context%20Challenge%20in%20Multicore.png)

### Data Driven Invocation

Module B in Core1 have an cyclic runnable which takes input value from shared memory process it and puts it back the output value to shared memory
