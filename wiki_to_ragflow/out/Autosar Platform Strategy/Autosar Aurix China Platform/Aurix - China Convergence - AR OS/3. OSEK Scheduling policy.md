## Introduction

OSEK supports two scheduling policy

- Full preemptive scheduling
- Non preemptive scheduling

### Non preemptive scheduling

Below is the example for Non preemptive scheduling,

Task T1 - High Priority - Non-Preemptive

Task T2 - Low Priority - Non-Preemptive

Task T1 activates when Task T2 is in running state, even though Task T1 has high priority it has to wait till Task T2 is suspended (Since Task T2 is a non-preemptive task)

![](../../../_images/3.%20OSEK%20Scheduling%20policy/image-2023-5-6_14-26-19.png)

### Preemptive scheduling

Below is the example for Preemptive scheduling,

Task T1 - High Priority - Preemptive

Task T2 - Low Priority - Preemptive

In this case, When Task T1 is activated, it preempts the Task T2 which is still running (Since Task T2 is preemptive)

![](../../../_images/3.%20OSEK%20Scheduling%20policy/image-2023-5-6_16-2-43.png)

> **INFO: GetResource or Lock ISRs**
> In the case of a full preemptive system, the user shall constantly expect preemption of the running task. If a task fragment shall not be preempted, this can be achieved by blocking the scheduler temporarily via the system service GetResource or Lock interrupts
